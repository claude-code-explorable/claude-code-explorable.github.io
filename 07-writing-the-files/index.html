Writing the files Claude will follow

The last chapter showed you which files exist. This one is the craft of writing them. Every line in CLAUDE.md is read at the start of every session and paid for on every turn — and past a point, a bloated file makes Claude ignore the rules that matter, because they are buried in the ones that don't. Here is a real, over-stuffed starter file. Give every line the right home, and watch the file that Claude will actually obey emerge.

Keep it in CLAUDE.md if…

  • It's a bash command or env quirk Claude can't guess.
  • It's a style or workflow rule that differs from the default.
  • It's concrete enough to verify — "2-space indent", not "format nicely".
  • It's needed in every session, everywhere in the repo.

Move it or cut it if…

  • Claude can read it off the code — package.json, the tree, git history.
  • It's a multi-step procedure → a skill, loaded on invoke.
  • It only matters for some files → a path-scoped rule.
  • It must happen every time → a hook; advice is only a request.
  • It's personal → CLAUDE.local.md, not the shared file.

How to read this. The draft opens the way most CLAUDE.md files actually grow — everything dumped in one always-on file. Click any line to inspect it, then pick its home. Keep the handful that earn their seat. Cut what Claude already knows or can read off the code. Send procedures to a skill, file-specific rules to a rule, must-happen guarantees to a hook, and personal notes to local. Watch context cost fall and adherence climb as the noise leaves — they are the same move seen twice. One line is deliberately vague; keeping it isn't enough, you have to make it specific first.
